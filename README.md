# 채용 자기소개서 분석 시스템

> 정부출연연구기관 의뢰로 구축한 자기소개서 기반 인재 평가 + 면접 질문 자동 생성 시스템.
> 두 갈래 NLP 파이프라인 — ① 임베딩 기반 정량 평가, ② LLM + RAG 기반 요약·질문 생성.
## 주의사항: `FILES` 경로는 `지원자데이터/` 폴더 기준으로 갱신됨 (set A=7번, set B=0-1·0-2번). 임베딩 캐시 키 `cache_embs_v5.pkl`로 bump.
---

## 개요

| 항목 | 내용 |
|------|------|
| 의뢰 기관 | 과학기술정책연구원 (STEPI) |
| 수행 소속 | 서강대학교 기술경영전문대학원(MoT) 연구실 (단기 인턴) |
| 참여 시기 | 2025 ~ 2026 |
| 담당 범위 | 데이터 파이프라인 · NLP 모델 · 평가 지표 설계 (단독 구현) |
| 비담당 | 백엔드 · 프론트엔드 |
| 운영 환경 | 온프레미스 (외부 API 사용 불가) / 24GB VRAM 단일 GPU |
| 코드·데이터 | 비공개 (의뢰 기관 자산). 본 문서는 방법론·의사결정 정리 |

자기소개서 텍스트를 입력하면 다음 세 가지를 자동 생성합니다.

1. **인재상 5축 정량 평가** — 임베딩 기반 유사도
2. **직무·부서 적합도** — 부서별 가중치 프로파일
3. **지원자 맞춤 면접 질문** — 과거 질문 DB RAG + LLM

---

## Highlights

- **임베딩 모델 검증 방법론** — SOTA 벤치마크가 아닌 *태스크 데이터 PCA 분산*을 기준으로 Jina vs BGE-M3 비교, Jina 채택
- **데이터 제약 환경의 모델 선택** — QLoRA 파인튜닝 시도 후 데이터 부족·문항 변동성을 이유로 미채택, Qwen3-14B (thinking) + 프롬프트 엔지니어링으로 전환
- **제약 조건이 만든 더 나은 설계** — 30B 단일 LLM 시도 후 *모델 크기와 무관하게 generic 질문만 생성*되는 한계 발견. 14B + RAG로 선회해 지원자 특성에 grounded된 질문 품질 확보
- **변별력 검증 시스템** — 축 가중치나 앵커 변경 시 `면접합격 > 서류합격 > 서류불합격` 순서를 자동 검증해 잘못된 튜닝 차단
- **데이터 mining 폐기, 도메인 정의 채택** (Decision 5) — anchor 자동 발견·정제 시도 모두 자기참조·신호 부족으로 실패. JD 핸드 앵커 그대로 사용하는 게 더 신뢰성 높다는 결론
- **임베딩 cosine → LLM 직접 채점** (Decision 6) — JD 추상 어조와 자소서 개인 서사 어조의 register 차이로 cosine 이 좁은 범위에 깔림. Qwen3-4B 1-10 채점 + 한 줄 근거로 변별 회복
- **RAG 면접 질문 생성** — 과거 질문 DB에서 유사 사례 retrieval 후 LLM이 지원자 프로필 반영해 맞춤 질문 생성

## Pipeline ① — 역량 평가 (3 metric, 다른 백엔드)

같은 5축 라벨을 공유하지만 *측정 신호*가 다른 세 metric. SPOF 회피 + 각 metric 의 task 특성에 맞는 백엔드 선택.

### #1 핵심인재 유사도 (cosine)

> "이 지원자 자소서가 *합격자 자소서 패턴*과 얼마나 닮았는가?"

```
자소서 문장 임베딩 (Jina v3, 1024-dim, 캐시)
        │
        ▼
직군별 합격자 centroid (축별 evidence centroid 평균) 와 cosine
        │
        ▼
Percentile 0-10 (전체 풀 884명 분포 기준)
```

descriptive cosine — 검증 로직 불필요. radar 한쪽 패널.

### #2 직무적합도 (LLM 직접 채점, Decision 6)

> "이 자소서가 STEPI 5축 정의에 *얼마나 부합하는가*?"

```
자소서 전문 + 5축 정의(JD 핸드 앵커) + 채점 기준
        │
        ▼
Qwen3-4B (4-bit) → JSON {축: {score 1-10, reason}}
        │
        ▼
한 줄 reason 면접관에게 노출 + cosine evidence 보조
```

옛 cosine 방식은 register 차이로 0.3~0.6 좁게 압축됨 (Decision 6). LLM 1-10 채점이 자연스러운 분포 + 채점 근거 제공.

### #3 직군 적합도 (rule-based)

> "이 지원자 5축 점수 패턴이 *어느 부서/직군 기대 프로파일*과 가장 가까운가?"

```
부서별 5축 기대 프로파일 (DEPT_PROFILES, HR 정의)
        │
        ▼
지원자 5축 점수 ↔ 프로파일 가중 유클리드 거리
        │
        ▼
부서별 적합도 점수 → bar chart
```

rule + 가중치만으로 충분. LLM 불필요.

### 핵심 설계 포인트

- **3개 metric 백엔드 직교** — cosine / LLM / rule 셋 다 다른 신호. SPOF 회피 + 사이드 검증 가능 (LLM ↔ cosine 괴리 시 flag).
- **상위 K 문장 선택** — #1 의 cosine 평균이 아닌 축과 가장 관련 깊은 문장만 평균. 노이즈 감소.
- **Percentile 정규화 (#1 만)** — 이상치에 강건. #2 는 LLM 출력 1-10 그대로, #3 은 거리 기반 그대로.
- **변별력 Flag (옛 검증)** — `면접합격 > 서류합격(면접탈락) > 서류불합격` 순서 자동 검증. 자소서로 합/불 변별이 본질적으로 약하다는 게 Empirical reality check 결과 (Decision 5/6 로 이어짐).

## Pipeline ② — 요약 + 면접 질문 생성

> "자소서를 LLM으로 구조화 요약하고, 그 요약을 기반으로 과거 면접 질문 사례를 참고해 맞춤 질문을 만든다."

### 처리 흐름

```
(1) 문항 의도 분석 → 서브카테고리 자동 생성
        │
        ▼
(2) 문항별 요약 (서브카테고리 기반, title/content 구조)
        │
        ▼
(3) 전체 요약 (3줄)
        │
        ▼
(4) 과거 면접 질문 DB 임베딩 + RAG 인덱스 (캐시)
        │
        ▼
(5) 문항별 요약 → 유사 질문 Top-K retrieval (직군 필터 적용)
        │
        ▼
(6) [지원자 프로필 + 요약 + 참고 질문 K개] → Qwen3-14B (thinking) 호출
        │
        ▼
(7) 맞춤 면접 질문 6~8개 생성 (JSON: 질문 + intent)
```

### 사용 모델

- **요약 LLM**: Qwen3-14B (4-bit 양자화, BitsAndBytes, **thinking 모드**)
- **질문 RAG 임베딩**: ko-sroberta-multitask
- **파인튜닝**: 보존만 (QLoRA 코드 유지, 미채택. 의사결정 근거는 아래 참조)

## 핵심 의사결정

### Decision 1 — 임베딩 모델: Jina vs BGE-M3

두 다국어 임베딩을 비교해 **jina-embeddings-v3 채택**.

**선택 기준**: PCA explained variance ratio

동일 자소서 문장 풀(수천 문장)을 두 모델로 임베딩한 뒤 PCA 적용. Jina 임베딩의 주성분 분산이 BGE-M3보다 일관되게 컸음.

**해석**:
- 분산이 크다 = 문장들이 임베딩 공간에서 더 잘 흩어져 있음 = 변별력 우위
- BGE-M3는 의미적으로 다른 문장도 비교적 가깝게 임베딩하는 경향 → 미묘한 의미 차이를 잡아야 하는 5축 매칭에서 약점

**검증 절차**:
1. 동일 문장 풀을 두 모델로 임베딩
2. PCA(2~10 components) 후 explained_variance_ratio 비교
3. 5축 앵커와의 코사인 유사도 분포 분산도 확인
4. Jina가 두 지표 모두에서 우위 → 채택

> **방법론적 교훈**: 임베딩 선택은 단순히 "최신 SOTA"가 아니라, *태스크 데이터에서의 변별력*으로 검증해야 한다. PCA 분산은 라벨 없이도 모델 품질을 가늠할 수 있는 좋은 프록시.

---

### Decision 2 — 요약 모델: 파인튜닝 vs 프롬프트 엔지니어링

3가지 접근을 시도하고 **Qwen3-14B (thinking) + 프롬프트 엔지니어링** 채택.

| # | 접근 | 결과 | 핵심 사유 |
|---|------|------|-----------|
| 1 | sLLM QLoRA 파인튜닝 | ❌ 미채택 | 데이터 부족 + 문항 변동성 |
| 2 | Qwen2.5-14B (thinking 없음) | ❌ 미채택 | 행간 추론 부족 |
| 3 | **Qwen3-14B (thinking 모드)** | ✅ 채택 | 행간 추론 + 요약 변별력 우수 |

**왜 파인튜닝을 포기했는가**:
- **문항 변동성**: 자소서 문항 자체가 채용 회차마다 변경됨. 파인튜닝 모델은 새 문항 형식에 즉시 일반화하기 어려움. 프롬프트는 문항 텍스트만 바꾸면 대응 가능.
- **데이터 부족**: 고품질 요약 페어 부족 → 과적합 위험 + 품질 보장 어려움.
- **운영 비용**: 모델 재학습 사이클 vs 프롬프트 수정 사이클의 비용 격차.

**왜 thinking 모드인가**:
- Qwen2.5는 표면적 요약은 가능했으나, 자소서의 모순점·공백·은유적 표현을 잡는 추론 깊이가 부족.
- thinking 모드가 동일 입력에 대해 더 깊은 행간 분석을 수행 → 요약 변별력·면접관 활용도 향상.
- 토큰 비용은 늘지만, 채용 도메인 특성상 *정확도 > 속도*.

> **교훈**: 데이터 제약 + 도메인 변동성 환경에서는 정교한 프롬프트가 파인튜닝보다 운영 효율적. 파인튜닝 코드는 데이터 누적 시 재시도 가능하도록 보존.

---

### Decision 3 — 역량 축 6 → 5: Cohen's d 기반 변별력 검증

초기 시스템은 6개 역량 축으로 설계됐으나, 변별력 분석 결과 1개 축이 약하게 나옴.

**검증 방법**:
- 면접합격군 vs 서류불합격군의 축별 점수 분포에 Cohen's d 적용
- 한 축에서 d 값이 다른 축 대비 현저히 낮음 (= 두 집단을 잘 구분하지 못함)

**원인 분석**:
- 해당 축은 자소서 텍스트로 직접 측정하기 어려운 추상적 개념
- 앵커 신뢰도가 다른 축 대비 낮았음

**조치**:
- 5축 시스템으로 축소
- 모든 축에서 변별력 Flag 통과 확인
- 추가로 anchor 단위 Cohen's d 분석으로 weak anchor 3개 제거

> **교훈**: 측정하고자 하는 개념이 데이터에 충분히 신호로 잡히는지 *수치로 검증*하는 단계가 필요. 직관적으로 "있어야 할 것 같은 축"이라도 변별력이 없으면 시스템 노이즈가 됨.

---

### Decision 4 — RAG 도입: "큰 모델"이 정답이 아니었던 이유

**초기 계획**은 30B급 단일 LLM에 자기소개서를 통째로 입력하고 면접 질문을 한 번에 생성시키는 단순 구조였음.

**제약 조건**:
- 운영 환경이 **온프레미스 + 24GB VRAM 단일 GPU**로 제한
- 외부 API 사용 불가 (의뢰 기관 정책)
- 30B 모델은 4-bit 양자화 후에도 컨텍스트·KV 캐시 포함 시 운영 불안정

→ 14B 모델로 다운사이즈하며 RAG 구조 도입 검토.

**그런데 다운사이즈 전, 30B 단일 LLM 방식을 짧게 시도했을 때 흥미로운 현상**:
지원자 자기소개서를 다 넣어줘도 LLM이 **모델 크기와 무관하게 generic 질문만 생성**. "강점은 무엇인가요?", "협업 경험을 말해주세요" 류의 질문이 지원자별로 거의 동일하게 나옴.

즉, 문제가 *모델 크기*가 아니라 *input grounding*에 있다는 발견.

→ **14B + RAG**로 선회.

**RAG 구조가 더 좋았던 이유**:

- **외부 grounding** — 과거 면접 질문 DB에서 유사 사례를 retrieval해 prompt에 포함. LLM이 추상적 질문에 빠지지 않고 실제 면접 패턴에 grounded
- **도메인 특이성** — 일반 LLM이 모르는 직군별 질문 결이 DB에 들어 있음. 14B여도 도메인에 맞는 질문 생성 가능
- **추적성** — Retrieval 결과를 로그로 남기므로 "왜 이 질문이 만들어졌는가"가 사후 추적 가능. 단일 LLM은 black box

> **교훈**: "더 큰 모델 = 더 좋은 결과"는 자동으로 성립하지 않는다. 면접 질문 생성처럼 *특정 입력에 grounded되어야 하는* 태스크에서는 모델 크기보다 입력 설계(retrieval, prompt structure)가 더 큰 변수. 자원 제약을 마주했을 때 단순히 모델을 줄이는 게 아니라, 시스템 구조 자체를 재설계할 때 오히려 결과가 좋아질 수 있음.

---

### Decision 5 — #2 직무적합도: 데이터 mining 폐기, JD 핸드 앵커 그대로

기존 설계는 "합격자 자소서에서 변별 신호 강한 문장을 mining해 anchor centroid 구성" 이었음. 다음 시도들이 모두 같은 약점에 부딪힘:

- KMeans(k=5) + K-fold deployment validation → fold 간 부호 뒤집힘 (5축이 자연 발견 안 됨)
- Qwen3-4B 5축 분류 후 Cohen's d top-K mining → 자기참조
- 우수 합격자 15명 자소서에서 핸드 앵커 ↔ 코사인 top-N 채택 → 자기참조 + 자소서 ≠ 합격 신호

**근본 원인**:
1. **합격자 자소서 ≠ 우수 자소서**. 합격은 자소서 + 전공·면접·경력 종합 판단의 결과. 자소서만으로는 합격 여부의 변별 신호가 약함 (Empirical reality check 가 보여준 결과).
2. **자기참조 회로**. 합격자 풀로 anchor를 만들고 그 anchor로 합격자를 평가하면 변별이 부풀려짐.
3. **JD/HR 정의가 ground truth**. 5축 정의를 데이터에서 *발견*할 수 없음 — 도메인이 *부여*해야 함.

**조치**: #2 anchor 는 HR/JD 가 직접 작성한 5축 × 6~7개 정의 문장 그대로 사용. 약한 anchor 문장은 HR이 다시 쓰면 됨, 데이터로 다듬을 게 아님.

> **교훈**: 데이터로 발견할 수 없는 신호도 있다. 데이터의 한계를 인정하고 도메인 지식을 직접 박아 넣는 결정이, 기술적 정교함보다 시스템 신뢰성에 더 기여할 수 있음.

---

### Decision 6 — #2 점수: cosine 폐기, LLM 직접 채점으로 전환

Decision 5 로 anchor 는 JD 그대로 가게 됐지만, 그 anchor 와 자소서 sentence 의 cosine 유사도로 점수를 매기니 **모든 지원자가 0.3~0.6 의 좁은 범위**에 깔림. 이유:

- JD anchor 는 HR 추상 어조 ("맡은 역할을 끝까지 수행하며…"), 자소서는 개인 서사 어조 — 의미는 가까워도 register 가 다름
- cosine 절대값이 좁아서 percentile 변환 시 노이즈에 극단적으로 민감
- 합격자조차 raw cosine 평균 0.4~0.5 → 변별 신호 자체가 압축됨

**조치**: #2 점수를 Qwen3-4B 직접 채점으로 교체.
- 입력: 자소서 전문 + 5축 정의 + 채점 기준 (1-3=일반론·의지, 4-6=언급은 있음, 7-8=구체 경험, 9-10=정량 성과)
- 출력: `{축: {score 1-10, reason 한 줄}}`
- LLM 이 1-10 분포를 자연스럽게 만들어 변별 회복, reason 한 줄로 면접관에게 채점 근거 제공

cosine 기반 근거문장 (top-K 자소서 sentence) 은 보조 자료로 유지 — LLM 채점이 *왜* 그 점수가 됐는지 텍스트 단위 추적 용도.

> **교훈**: 임베딩 cosine 은 "의미적 유사도"의 한 측정일 뿐이고, 두 텍스트 register 가 다르면 절대값이 좁게 깔린다. 이 한계가 명확해진 시점에 LLM 직접 채점으로 갈아타는 게 정직한 선택.

---

### Decision 7 — Pipeline ② RAG 유지 (파인튜닝 교체 안 함)

**계기**: 채용 담당자 인터뷰 결과 *면접 시 사용한 질문을 더 이상 기록하지 않는다*. 질문 DB 동결 (~1,233건). 그러면 파인튜닝이 RAG보다 낫지 않냐는 재검토.

**결론: RAG 유지**. 다음 4 가지 이유:

1. **DB 동결이 RAG 약점이 아님**. RAG 가치는 DB 신선도가 아니라 *per-applicant retrieval* 자체. DB 가 동결돼도 지원자마다 다른 Top-20 이 검색되어 나옴 — Decision 4 에서 generic 질문 막은 메커니즘은 그대로 작동.
2. **데이터 부족은 그대로**. Decision 2 의 파인튜닝 거절 근거 = 데이터 부족 + 문항 변동성. DB 동결돼도 양 변화 없음.
3. **Generic 함정 재현 위험**. 파인튜닝은 지식 베이크지 grounding 메커니즘이 아님. Decision 4 의 30B 단일 LLM 실패 패턴 재현 가능.
4. **운영비·추적성 RAG 우세**. 직군 신설·역할 변경 시 RAG는 DB 1줄 추가, 파인튜닝은 풀 재학습. 추적성도 retrieval log 로 즉시.

**조치 (DB 동결 부작용 완화)**:
- Retrieval 품질 모니터링 (Top-K 평균 코사인 < threshold 인 지원자 비율 분기 점검)
- DB 정제 1회 (generic 질문 / 폐기 직군 질문 제거)
- 신규 직군/토픽 합성 augment 옵션
- 스타일 SFT 검토 (어조 일관성용 가벼운 LoRA, content grounding 은 RAG 유지)

> **교훈**: 외부 환경 변화가 기존 설계를 흔드는 *직관*이 들어도, 결정의 *원래 근거*가 변하지 않았다면 결정도 변할 이유가 없다. "신선한 데이터" 와 "검색 다양성" 은 다른 개념이며, 후자는 DB 가 동결돼도 유지된다.

## Lessons Learned (도메인 무관)

### 1. 임베딩 재계산 비용 → pickle 캐시 + 버전 키
1024-dim × 수천 문장 × 매 실행마다 재계산은 자원 낭비. 임베딩 캐시 파일을 버전 키와 함께 관리해 데이터 변경 시에만 무효화.

### 2. LLM JSON 출력 안정성
LLM이 마크다운 code fence(\`\`\`json ... \`\`\`)나 앞뒤 설명을 붙이는 경향. 시스템 프롬프트에 "마크다운 없이 JSON만 출력" 명시 + 파서에서 정규식으로 JSON 블록 추출하는 이중 방어.

### 3. 핵심 ID 검증 패턴
gold-standard ID를 코드에 하드코딩한 뒤 데이터 갱신으로 일부가 누락되는 사고. 발견 카운트가 0이면 명시적 에러를 던지는 검증 단계로 silent failure 차단.

### 4. 데이터 join 키 부재 시의 fallback
서로 다른 출처의 데이터 파일에 공통 키가 없을 때, 행 순서 가정 기반 합성 ID를 부여. 단 정렬 어긋나면 매칭이 통째로 망가지므로 정렬 기준 검증을 반드시 동반.

### 5. 변별력 Flag 같은 자동 검증을 시스템에 내장
사람이 결과를 보고 "이상한데?"를 사후에 잡는 게 아니라, 튜닝 단계에서 잘못된 변경을 자동으로 차단하는 게이트가 있어야 함. 모델·앵커 변경처럼 결과가 미묘하게 망가지는 변경에서 특히 효과.

## Risk Register

LLM 의존이 task 특성상 불가피한 만큼(텍스트 요약·생성·평가에서 임베딩으로 대체 불가), *모델 교체* 가 아니라 *방어선* 으로 위험을 관리하는 설계.

### LLM 의존 위험 + 완화책

| 위험 | 완화 |
|---|---|
| **재현성 약함** — 모델·프롬프트 변경 시 점수 비교 의미 상실 | 배치마다 모델 fingerprint, transformers 버전, 프롬프트 버전, 생성시각 자동 로깅 (`output/llm_score_metadata.json`). 운영자가 비교 시 메타 일치 확인 |
| **단일 모델 SPOF** — LLM 실패 시 4/6 컴포넌트 죽음 | #1 핵심인재 유사도(임베딩 cosine), #3 직군 적합도(rule-based) 가 LLM-free 라 부분 작동 보장. 설계 단계에서 차단 |
| **블랙박스 채점** — 점수 근거가 LLM reason 한 줄에 의존 | UI에 LLM reason + cosine evidence 문장을 함께 노출. 면접관이 양쪽 보고 sanity check |
| **Calibration drift** — 합격자 풀 평균이 분기마다 흔들리면 점수 기준 무너짐 | 분기별 calibration 스냅샷 (축별 mean/std, ±0.7/±0.5 초과 시 경고). 모델·프롬프트 버전 변경도 자동 감지 |
| **LLM ↔ cosine 괴리** — LLM이 cosine과 크게 다른 점수 매기면 신뢰 어려움 | render 시 LLM 1-10 vs cosine percentile 10pt 차이 > 4 축 표시: "코사인 유사도와 LLM 기반 점수의 괴리가 크니 근거문장 확인 요망" |

### 데이터 위험 + 완화책

| 위험 | 완화 |
|---|---|
| **일부 코호트 자소서 본문 부재 / 라벨 NaN** | 통합 로더가 자동 누락 처리, 요약에 표시 |
| **DB 동결로 신규 직군 retrieval miss 가능 (Decision 7)** | 분기별 retrieval 평균 cosine 모니터링 + DB 정제 1회 + 합성 augment 옵션 |
